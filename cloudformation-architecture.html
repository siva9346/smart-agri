What sam deploy actually does — step by step

1

You declare the desired state

backend/template.yaml says what should exist — not how to create it. The Transform: AWS::Serverless-2016-10-31 line at the top is what turns this from a plain CloudFormation template into a SAM template.

2

sam build packages the code

For each function, SAM installs requirements.txt against the matching python3.12 runtime and copies your handler source into .aws-sam/build — one self-contained artifact per Lambda.

3

sam deploy uploads artifacts to S3

Each function's build is zipped and uploaded to a SAM-managed bucket (aws-sam-cli-managed-default-samclisourcebucket-…), deduplicated by content hash — this is why re-deploying unchanged code just says File with same data already exists, skipping upload.

4

The packaged template is submitted to CloudFormation

SAM CLI rewrites every CodeUri to point at the uploaded S3 object, then calls CreateChangeSet — still with the shorthand AWS::Serverless::Function / AWS::Serverless::HttpApi resources intact.

5

CloudFormation expands the SAM transform

Before diffing anything, CloudFormation runs the SAM macro server-side: each Function becomes a real Lambda::Function + a generated IAM::Role + a Lambda::Permission; the single Api resource becomes an ApiGatewayV2::Api with all 11 routes baked into its auto-generated OpenAPI document.

6

The change set is computed

CloudFormation diffs the expanded template against the current stack. On our first deploy that meant 50+ lines of + Add — every table, role, function, and permission. On a future code-only change, this step would show just one Modify for the function that changed, nothing for the untouched tables.

7

The change set executes, in dependency order

DynamoDB tables and IAM roles first (nothing else depends on them existing), then Lambda functions (need their role + S3 code), then the permissions and routes that let API Gateway actually reach them. Every resource is tracked CREATE_IN_PROGRESS → CREATE_COMPLETE individually — if any one fails, the whole stack automatically rolls back.

8

Outputs resolve, stack goes green

Once every resource reports complete, the stack itself flips to CREATE_COMPLETE and the Outputs block evaluates — ApiUrl is a !Sub that reads the real, just-created API's ID: oe2hznr4xi.

9

The frontend catches up

scripts/sync-api-url.sh reads that same Output back out via describe-stacks and writes it straight into src/config.ts — closing the loop from infrastructure back to the app you ship.

Continuous deployment — the same process, triggered by git

.github/workflows/deploy.yml runs on every push to master that touches backend/**. It does exactly steps 1–9 above, just from a GitHub-hosted runner instead of your terminal: it assumes naveena-uzhavan-github-deploy via OpenID Connect (no long-lived AWS keys stored in GitHub), then runs the identical sam build && sam deploy. CloudFormation doesn't know or care whether the request came from your Mac or a GitHub runner — it diffs the template against the live stack either way.

The one manual step left: add AWS_DEPLOY_ROLE_ARN as a repository secret on siva9346/smart-agri, set to arn:aws:iam::964604399680:role/naveena-uzhavan-github-deploy.
